RADOS 는 Ceph 의 지원 구성 요소이며, Ceph 의 현재 3 가지 핵심 응용 프로그램 구성 요소 인 RBD, RGW 및 CephFS (각각 블록, 오브젝트 및 파일시스템 인터페이스 제공) 외에도 RADOS 및 파생 된 Librados 표준 라이브러리를 기반으로 합니다. 다른 유형의 다른 응용 프로그램 구성 요소를 개발할 수도 있습니다. 이 책은 RADOS와 RBD, RGW 및 CephFS의 세 가지 핵심 응용 프로그램 구성 요소에 중점을두고 있습니다. 자세한 장과 소개는 다음과 같습니다.

1장 왕으로 계산(?) - CRUSH

CRUSH는 Ceph의 두 가지 핵심 디자인 중 하나입니다. CRUSH 의 우수한 설계 개념은 컴퓨팅 어드레싱, 높은 동시성 및 동적 데이터 밸런싱과 같은 기본 기능과 분산화와 같은 구현이 매우 편리하고 물리적 구조 변경을 효과적으로 방지하며 성능이 클러스터 크기에 따라 달라지는 맞춤형 복제 전략을 갖도록합니다.과

선형 확장 및 높은 신뢰성과 같은 고급 기능을 갖추고 있으므로 확장성, 성능 및 안정성에 대한 엄격한 요구 사항이 있는 Ceph 과 같은 대규모 분산 스토리지 시스템에 매우 적합합니다.

2장 최고의 성능

Jewel 릴리즈 이후, 커뮤니티는 10년 이상 서비스를 제공한 Filestore 를 대체하기 위해 새로운 유형의 고성능 오브젝트 스토리지 엔진인 BlueStore 를 도입했습니다. BlueStore 의 도입은 의심할 여지 없이 지난 2년 동안 Ceph 의 가장 매력적인 기능 중 하나입니다.

3장 시공간 게임(?) - Erasure Code

eph 의 기존 3 카피 데이터 백업 방법은 높은 신뢰성을 달성한다는 전제에서 프론트엔드 요청의 응답 지연을 최소화 할 수 있으므로 안정성과 성능에 대한 요구사항이 있는 응용 프로그램에 적합합니다. 가장 널리 사용되는 3 카피 방법의 단점은 추가 스토리지 공간을 많이 차지하여 클러스터의 실제 공간 활용도가 낮아진다는 점입니다.

반대로, Erasure Code 는 Strip 을 단위로 사용하며, 수학적 연산을 통해 k + m 백업 전략을 사용합니다. 지연 시간이 길기 때문에 지연에 민감하지 않는 "콜드 데이터" (예: 백업 데이터) 응용 프로그램에 적합합니다. Kraken 버전에서 커뮤니티는 처음으로 Erasure Coding 에서 가장 복잡한 덮어쓰기 문제를 해결하여 프로덕션 환경 시대의 시작을 보였습니다.

4장 이주의 아름다움(?) - PG

PG 는 Ceph 의 가장 핵심적이고 복잡한 개념 중 하나이며, Ceph 의 가장 어려운 과제 중 하나입니다. PG 의 뛰어난 기능 중 가장 중요하고 눈에 띄는 것은, CRUSH 의 실시간 계산 결과를 기반으로 OSD 간에 자유룝게 마이그레이션 할 수 있다는 것입니다. 이는 Ceph 자동 데이터 복구 및 균형 유지를 할 수 있게 합니다.

5장 선행 제어(?) - QoS

가상화 기술이 인기를 얻고 있으며 제한된 리소스를 중앙에서 관리하고 필요할 때 할당하여 효율을 극대화하는 방법은 항상 주요 주제 중 하나였습니다. QoS 기능을 적극적으로 도입함으로서 Ceph 은 IOPS 및 클러스터 대역폭과 같은 I/O 리소스를 합리적으로 조정함으로서 외부에 보다 세련된 스토리지를 제공합니다.

6장 실수로(?)

Sage A. Weil 이 2007년 공식적으로 Ceph 을 출시한 이후 Ceph 은 10년 이상 개발되었습니다. 설계 초기에 Ceph 은 순수 분산 파일 시스템 (Cephfs) 로 자리 잡았지만, 가상화가 점차 정보화 시대의 주제가 되었고, Openstack 이 대표하는 클라우드 컴퓨팅 기술의 등장으로 커뮤니티는 초점을 단호히 결정하고 개발에 집중하기 시작했습니다.

새로운 유형의 분산 블록 스토리지 서비스 구성요소인 RBD 는 점차 가상머신, 미러링, 클라우드 디스크 및 Openstack 과 같은 IaaS 클라우드 컴퓨팅 환경의 기타 서비스에 없어서는 안될 기본 블록 디바이스 스토리지 백엔드가 되었습니다. Ceph 은 많은 유사한 소프트웨어의 경쟁에서 두드러질 수 있었으며, Openstack 의 성장 덕분에 점차 가장 인기있는 분산 통합 스토리지 시스템으로 성장할 수 있었습니다.

7장 클라우드

Wu Jun 박사는 "Top of the Tide" 저서의 서문에서 다음과 같이 말했습니다. "100년이 넘는 기간동안 일부 회사는 기술 혁신의 끝에 서서 다행스럽게도 의식적/무의식적으로 일해 왔습니다. 10년 동안, 그들은 다음 물결까지 기술의 물결을 대표합니다(?)"

현재 떠오르는 클라우드 컴퓨팅은 차세대 기술 개발의 도래를 나타내고 있으며, AWS 를 기반으로 퍼블릭 클라우드 서비스를 처음 시작하고 퍼블릭 클라우드의 사실상 표준이 된 Amazon 은 의심의 여지 없이 물결의 선두에 서있습니다. 실제로, 2006년 도입된 이래 AWS 에 저장된 객체의 현재 규모는 천억 수준에 이르렀으며, 아마존에 100억 달러 이상의 수익이 축적되어 클라우드 컴퓨팅에 포함된 엄청난 비지니스 기회를 보여줍니다. AWS 는 스토리지 시스템이 기존 블록 및 파일 스토리지 (오브젝트 스토리지 인터페이스) 와 다른 세 번째 유형의 인터페이스를 제공할 수 있어야 하며, 사용자 정의 S3 프로토콜을 사용하여 HTTP 를 통해 전송합니다. 이러한 맥락에서 클라우드 컴퓨팅으로 표현된 이 기술의 물결을 따라 잡기 위해 Ceph 은 클라우드에서 생성된 S3-RGW 로 표현된 Object Storage Protocol Cluster의 Object Storage Gateway와 호환됩니다.

8장 클래식 재현

파일시스템은 운영체제와 함께 탄생했으며, 컴퓨터 공학에서 가장 기본적이고 고전적인 개념 중 하나입니다. Ceph 은 탄생 이래 분산 파일 시스템으로서 자리매김했습니다.

Ceph 의 세 가지 일반적인 어플리케이션 시나리오에서 RBD 와 RGW 는 클라우드 컴퓨팅을 주도적으로 발전시켰으며, 나중에는 점점 인기를 얻었지만, 가장 초기의 CephFS 는 속도를 늦추었습니다. 첫번째 이유는 파일 시스템이 트리 구조를 사용하여 데이터 (파일 및 디렉토리) 를 관리하고, 주소를 지정하는 플랫 방법을 사용하는 Ceph 의 설계 개념과 호환되지 않았기 때문입니다. 파일 시스템을 지원하려면 Ceph 이 중앙 집중식 메타데이터 관리 서버 (어드레싱을 위한 트리 구조의 통합된 엔트리포인트) 를 도입해야 하며, 이는 거의 무제한적인 수평 확장성을 추구하는 Ceph 의 분산 설계 철학과 충돌합니다.

부인할 수 없는 사실 중 하나는 RBD 및 RGW 의 급격한 발전으로 인해 클라우드 컴퓨팅 이외의 영역에서 Ceph 이 급속히 대중화되어 점차 널리 알려지게 되었다는 것입니다. 업계에서는 기존 블록 및 파일 스토리지 장비가 감소함에 따라 Ceph 은 진정한 통합 스토리지 시스템으로서 기존 스토리지를 흡수할 수 있을 것으로 기대합니다. 따라서 긴 파일 장애에도 불구하고 기존 파일 스토리지를 교체하는데 중요한 부분으로 CephFS 리서치를 다시 시작하고, 가능한 빨리 프로덕션 환경에 배치하고 있습니다.

9장 사용의 마술

한마음으로 사용하십시오. Ceph의 기본 원칙을 배우는 데 오랜 시간을 보낸 후, 나는 대부분의 독자가 실습을 통해 Ceph의 마법을 경험할 수 있으리라 믿습니다. 이 책의 끝에서, 우리는 프로덕션 환경에서 사용되는 다양한 Ceph 운영을 독자들을위해 신중하게 준비했습니다.
